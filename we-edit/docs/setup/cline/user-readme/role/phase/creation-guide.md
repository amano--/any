# フェーズ別資料集作成の手順書

## 1. 基本的なファイル構成と特徴

### 1.1 標準ディレクトリ構成
```
phase-name/
├── README.md                                     # フェーズの概要と資料の使用方法
├── experts-list-[category]-[number].md          # 各カテゴリの専門家リスト
├── role-usage-guide.md                          # ロールの具体的な活用方法
├── case-studies.md                              # 実際の活用事例集
├── quick-reference.md                           # クイックリファレンス
├── troubleshooting.md                           # よくある課題と解決策
└── additional-resources/                        # 追加リソース
    ├── templates/                               # 各種テンプレート
    ├── workshops/                               # ワークショップガイド
    └── examples/                                # 具体的な例
```

### 1.2 各フェーズの特徴的な構成例

#### 要求分析フェーズ
```
requirements-analysis/
├── experts-list-methodology-01.md               # 要求工学の専門家
├── experts-list-domain-01.md                    # ドメイン分析の専門家
├── templates/
│   ├── requirement-specification.md             # 要求仕様書テンプレート
│   ├── user-story.md                           # ユーザーストーリーテンプレート
│   └── acceptance-criteria.md                   # 受け入れ基準テンプレート
```

#### 設計フェーズ
```
design-phase/
├── experts-list-architecture-01.md              # アーキテクチャの専門家
├── experts-list-patterns-01.md                  # パターンの専門家
├── templates/
│   ├── architecture-design.md                   # アーキテクチャ設計書
│   ├── component-spec.md                        # コンポーネント仕様書
│   └── interface-design.md                      # インターフェース設計書
```

#### 実装フェーズ
```
implementation/
├── experts-list-development-01.md               # 開発手法の専門家
├── experts-list-quality-01.md                   # 品質管理の専門家
├── templates/
│   ├── coding-guidelines.md                     # コーディングガイドライン
│   ├── test-plan.md                            # テスト計画書
│   └── review-checklist.md                      # レビューチェックリスト
```

### 1.3 ファイルの命名規則

#### 専門家リストの命名
```
experts-list-[category]-[number].md

カテゴリ例：
- methodology    # 方法論
- domain        # ドメイン知識
- architecture  # アーキテクチャ
- development   # 開発手法
- quality       # 品質管理
- process       # プロセス改善

番号付け：
- 01から開始
- 1ファイルあたり4-5名が目安
- カテゴリ内で複数ファイル可
```

#### テンプレートの命名
```
[purpose]-[type]-[variant].md

例：
- requirement-specification-basic.md
- architecture-design-detailed.md
- test-plan-comprehensive.md
```

## 2. 専門家リストの作成手順

### 2.1 専門家の選定基準
```markdown
### 必須要件
- 分野での重要な貢献
- 実践的な手法の確立
- 具体的なプラクティス
- 検証可能な成果

### 推奨要件
- 複数の成功事例
- 体系的な方法論
- 実装可能なフレームワーク
- 継続的な影響力
```

### 2.2 専門家情報の構造化
```markdown
1. プロフィールセクション
   - 専門分野の定義
   - 主要な貢献
   - 代表的な著作
   - 影響力の範囲

2. 活用シナリオ
   - 適用場面
   - 期待される効果
   - 制約条件
   - 成功事例

3. プロンプト例
   - 基本構造
   - バリエーション
   - カスタマイズポイント
   - 使用上の注意
```

## 3. テンプレートの設計と作成

### 3.1 基本構造
```markdown
### ヘッダー情報
- ドキュメント識別子
- 作成日/更新日
- バージョン
- ステータス

### コンテンツ構成
1. 目的と範囲
2. 前提条件
3. 主要セクション
4. 補足情報

### フッター情報
- 参照情報
- 変更履歴
- レビュー記録
```

### 3.2 品質基準
```markdown
### 必須要素
□ 目的の明確さ
□ 構造の一貫性
□ 用語の統一
□ 参照の明確さ

### 推奨要素
□ 具体例の提供
□ カスタマイズ方法
□ チェックリスト
□ トラブルシューティング
```

## 4. ロール活用ガイドの作成

### 4.1 基本構成
```markdown
1. ロールの概要
   - 専門性の定義
   - 主要な貢献
   - 適用範囲

2. 活用シナリオ
   - ユースケース
   - 具体的な手順
   - 期待される成果

3. プロンプト例
   - テンプレート
   - カスタマイズ方法
   - 注意点
```

### 4.2 実践的な活用方法
```markdown
### シナリオベースの構成
1. 状況設定
   - プロジェクトフェーズ
   - 課題の性質
   - リソース制約

2. アプローチ選択
   - 適切な専門家
   - 手法の組み合わせ
   - 実施手順

3. 成果物
   - 期待される output
   - 品質基準
   - レビューポイント
```

## 5. 事例集の作成

### 5.1 事例の構造
```markdown
### 基本情報
- プロジェクト概要
- 課題の性質
- 適用したアプローチ

### 実施内容
1. アプローチの選択理由
2. 具体的な実施手順
3. 遭遇した課題
4. 解決策

### 結果と学び
- 成功/失敗の要因
- 得られた知見
- 改善点
```

### 5.2 効果的な事例選択
```markdown
### 選択基準
□ 一般性の高さ
□ 具体的な手順の明確さ
□ 結果の検証可能性
□ 学びの普遍性

### カバレッジ
□ 異なる状況
□ 様々なアプローチ
□ 成功/失敗のバランス
□ 多様な課題
```

## 6. メンテナンスと改善

### 6.1 定期レビュー
```markdown
### レビュー項目
1. 内容の正確性
2. 最新性の確認
3. 実用性の評価
4. フィードバックの反映

### 更新サイクル
- 月次：軽微な更新
- 四半期：内容の見直し
- 半期：大規模改訂
```

### 6.2 品質管理
```markdown
### チェックポイント
□ 構造の一貫性
□ 内容の正確性
□ リンクの有効性
□ 用語の統一性

### 改善プロセス
1. フィードバック収集
2. 改善点の特定
3. 更新計画の立案
4. 変更の実施
```

## 7. プロジェクトの状況に応じた調整

### 7.1 プロジェクト規模による調整
```markdown
### 小規模プロジェクト
- 必須ドキュメントの特定
- テンプレートの簡略化
- プロセスの軽量化

### 大規模プロジェクト
- 詳細な文書化
- 厳密なレビュー
- 包括的な管理
```

### 7.2 チーム状況による調整
```markdown
### 経験レベルに応じた調整
- 初級者向けガイド追加
- 詳細な例示の提供
- サポート体制の強化

### 分散チームへの対応
- オンライン協業ツール
- 非同期コミュニケーション
- 文書の共有方法
```

### 定量的指標

### プロセス
1. データ収集

3. 改善点の特定
4. 施策の実施
